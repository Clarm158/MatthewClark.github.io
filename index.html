00 / self-assessment

Professional Self-Assessment

Completing the Computer Science program at Southern New Hampshire University has been a way for me to strengthen the technical side of the work I already do. I have spent the last several years working in flight safety and aircraft simulation, where I interact with complex systems on a regular basis. More recently, I transitioned into a project management role, which added another layer of responsibility in coordinating technical and operational teams. This degree was not about changing careers, but about building a better understanding of how the software behind these systems actually works and how to make more informed decisions because of it.

Strengths and Professional Growth

One of the biggest changes for me throughout this program was how I think about code. Early on, my focus was just getting things to work. Over time, especially through courses like CS 260 and CS 320, I started paying more attention to how code is structured, how easy it is to understand, and whether it would hold up if someone else had to work on it later. That shift from working code to maintainable code is something that directly connects to the kind of systems I deal with professionally.

Collaboration has always been a big part of my job, and that carried over into the program. Courses like CS 250 and IT 315 helped reinforce structured approaches to working with others, documenting decisions, and breaking work into manageable pieces. These were things I was already doing in practice, but the program gave me a clearer framework for them and helped me communicate those ideas more effectively.

Another area that improved was how I explain technical work. Whether it is writing documentation, explaining design decisions, or walking someone through a problem, being clear and direct matters. CS 499 pushed me to not just show what I built, but explain why I made certain decisions and what tradeoffs were involved, which is very similar to what I do when communicating with stakeholders in my current role.

Data Structures and Algorithms

My coursework in data structures and algorithms gave me a practical vocabulary for evaluating computational efficiency. Working with sorting algorithms, hash tables, and vectors in C++ helped me understand how the choice of data structure affects performance at scale. The selection sort, quicksort, and merge sort implementations I developed reinforced that algorithm selection is a design decision, not an afterthought. Understanding Big-O complexity and how pivot selection in quicksort affects partition balance are the kinds of details that separate functional solutions from well-engineered ones.

Software Engineering and Databases

Across the program, I developed a clearer sense of what good software engineering practice looks like at the design level. CS 330 (Computational Graphics and Visualization) pushed me into unfamiliar territory with OpenGL, teaching me how to reason about rendering pipelines and coordinate systems. On the database side, working with both SQL and MongoDB in CS 340 (Advanced Programming Concepts) gave me experience with two very different data models and helped me understand when each is appropriate. Building and debugging a Dash-based dashboard backed by MongoDB reinforced how front-end and back-end layers interact and how poorly structured callback logic can cascade into broader system failures.

Security

Security has been a thread running through multiple courses, and it is an area I approach with particular seriousness given my background. The Artemis Financial project in CS 405 was one of the most directly applicable experiences in the program. Working through encryption, HTTPS configuration, certificate management, and dependency vulnerability scanning reinforced the principle that security is not a feature added at the end — it is a property that must be designed in from the beginning. That perspective shapes how I evaluate software decisions in my professional context as well.

Throughout the program, I developed skills across several key areas that directly apply to my professional work. From a collaboration standpoint, I regularly worked in structured environments that emphasized communication and shared responsibility. This aligns closely with my current role, where I coordinate across teams and stakeholders. In terms of data structures and algorithms, I gained a better understanding of how design decisions impact performance, especially when working with sorting algorithms and data organization. My experience with software engineering and databases, particularly through building applications and working with MongoDB, helped me understand how different system components interact. Finally, security concepts reinforced the importance of building systems that are reliable and protected from the start, which is especially relevant in safety-critical environments like aviation.

How the Artifacts Fit Together

The three enhancements in this portfolio were selected to demonstrate growth across software design and engineering, algorithms and data structures, and databases. The first two enhancements work with the same C++ bid-sorting program — an honest early-stage artifact that was functional but rough in its structure. Enhancement One addresses design and engineering through refactoring and input validation. Enhancement Two expands the algorithmic scope by adding merge sort alongside the existing methods. Enhancement Three introduces a Python-based dashboard backed by MongoDB, representing a more complete system that required debugging, multi-criteria filtering, and callback synchronization. Together, these artifacts tell a coherent story — from low-level algorithmic work to full-stack application development — with software design principles threading through all three.

Artifact Enhancements

Enhancement 01
Software Design & Engineering
C++ Bid Sorter — Refactoring & Input Validation
+

ARTIFACT DESCRIPTION

A C++ program developed early in the program for working with a dataset of auction bids. Users can load data from a CSV file, display bids, and sort using selection sort or quicksort. The original was functional but lacked structure and separation of responsibilities.

WHAT WAS IMPROVED

The main focus of this enhancement was cleaning up the structure of the program without rewriting it completely. I separated out parts of the logic, like menu handling and data display, so the flow of the program was easier to follow. I also added input validation to prevent invalid entries from causing issues, which was something the original version did not handle well.

This enhancement was included to demonstrate my ability to improve software design without rebuilding a project from scratch, which reflects growth in software engineering practices.

SKILLS DEMONSTRATED

Separation of concerns, function decomposition, input validation, and applying software design principles to improve an existing codebase without completely rewriting it.

REFLECTION

This enhancement showed how much my approach to coding has changed. Originally, I was focused on just getting the program to run. Going back to it, I was thinking more about readability, structure, and how someone else would understand it. The biggest challenge was improving the code without overcomplicating it or turning it into a completely new project.

Enhancement 02
Algorithms & Data Structures
C++ Bid Sorter — Algorithm Improvement
+

ARTIFACT DESCRIPTION

The same C++ bid-sorting program, now extended to include a third sorting algorithm. The original was limited to selection sort and quicksort, restricting the ability to compare different algorithmic approaches on the same dataset.

WHAT WAS IMPROVED

This enhancement focused on improving how sorting was handled and expanding the program’s ability to compare different approaches. Instead of just relying on the existing implementations, I looked at how the algorithms performed and made adjustments to improve efficiency and clarity. This also helped reinforce how different sorting strategies behave under the same dataset.

This enhancement was selected to demonstrate growth in algorithms and data structures by focusing on how different approaches affect performance and efficiency.

SKILLS DEMONSTRATED

Algorithm analysis, comparative evaluation of sorting behavior, integration of new logic into an existing codebase, and understanding how implementation choices affect efficiency and scalability.

REFLECTION

Working on this part made it clear that small decisions in algorithms can have a big impact on performance. Things like how data is divided or how comparisons are handled are easy to overlook when everything seems to work, but they make a difference as the dataset grows. It also reinforced that understanding the behavior of an algorithm is just as important as implementing it.

Enhancement 03
Databases
Python / MongoDB Dashboard — Filtering & Map Sync
+

ARTIFACT DESCRIPTION

A Python-based Dash dashboard for visualizing animal shelter data stored in MongoDB. The original supported filtering by rescue category (Water, Mountain, Disaster) but had inconsistent filtering behavior and a geolocation map that did not update correctly when records were selected.

WHAT WAS IMPROVED

The main improvements focused on making the dashboard more usable and fixing issues with how the data was being handled. I expanded the filtering system so users could filter by multiple criteria instead of just predefined categories. I also fixed the map so it updates correctly when a record is selected, which was not working consistently in the original version.

This artifact was included to demonstrate my ability to work with databases and integrate them into a functional application, showing how data, logic, and user interface components interact.

SKILLS DEMONSTRATED

MongoDB query construction, Dash callback architecture, dynamic UI state management, debugging data synchronization across components, and integration between the database, application logic, and visual output.

REFLECTION

This was one of the more challenging parts because the issue was not obvious at first. The data itself was correct, but the way it moved through the application caused the map to fall out of sync. Fixing it required stepping back and looking at how everything was connected instead of focusing on just one part of the code.
